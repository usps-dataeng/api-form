üìπ Storage Blob Logging Explained

üö® The "Security Camera Problem"

Imagine security cameras that record every person walking by, every car passing, every bird flying instead of just recording break-ins and burglaries. Then storing 2 years of useless footage. This is what StorageBlobLogs does - records every file read/write instead of just errors and security events.

üìπ What StorageBlobLogs Records

  • Every file read - Like recording who looked at each document
  • Every file write - Like filming every time someone files a paper
  • Every folder browse - Like recording every time someone opens a filing cabinet
  • System operations - Like filming the cleaning crew organizing files

üîç In Our Case:

  • 287 GB/day of "security footage" (logs)
  • Databricks + applications = constant file activity
  • High-volume storage = massive log generation

üéØ Why This Happened

  • Default settings: Azure enables detailed logging by default
  • Nobody monitoring: Logs growing without oversight
  • Retention already optimal: 30 days (no change needed)
  • Wrong level: INFO/DEBUG vs. ERROR-only needed

üí° The Fix:

  • Stop recording everything (disable read/write logging of normal activity)
  • Only record actual security events (enable delete/error/security logging only)
  • Retention already optimal (30 days is perfect)
  • ERROR-only recording (problems only, not everything)

Unlike traditional fixed costs, Azure Pay-As-You-Go billing means every action has a cost that compounds over time.

  • Large scale: TB of data movement daily

üí∏ What Drove High Costs:

  • Every API call = monitoring cost

üß™ Dev/Test Environment

  • Lower-volume operations: Limited usage patterns
  • Intermittent activity: Only during development
  • Smaller scale: GB vs. TB data movement
  • Result: Lower but still unnecessary costs
